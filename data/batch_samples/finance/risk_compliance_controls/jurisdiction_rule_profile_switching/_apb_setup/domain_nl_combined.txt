DOMAIN IN NATURAL LANGUAGE

--- TYPE HIERARCHY ---

  Everything that is a case instance or a profile candidate or a rule set or a account attribute or a customer segment or a transaction attribute or a profile source or a escalation ticket or a approval policy or a senior reviewer or a subject matter expert or a jurisdiction code or a escalation handle is also a object
  Everything that is a escalation group or a escalation team is also a escalation handle
  Everything that is a case category a or a case category b is also a case instance

--- ACTIONS ---

  register a case instance A
    I can only register a case instance A if it is not the case that A is registered and it is not the case that A has committed a profile switch
    Once I register a case instance A, it is the case that A is registered

  select a profile candidate B for a case instance A
    I can only select a profile candidate B for a case instance A if it is the case that A is registered and B is available and A is compatible with B
    I can only select a profile candidate B for a case instance A if it is not the case that the candidate for A is bound
    Once I select a profile candidate B for a case instance A, it is the case that A has the candidate profile B selected and the candidate for A is bound
    Once I select a profile candidate B for a case instance A, it is not the case anymore that B is available

  release a profile candidate B from a case instance A
    I can only release a profile candidate B from a case instance A if it is the case that A has the candidate profile B selected
    I can only release a profile candidate B from a case instance A if it is not the case that A has case rules applied flag and it is not the case that A has case approval initiated
    Once I release a profile candidate B from a case instance A, it is the case that B is available
    Once I release a profile candidate B from a case instance A, it is not the case anymore that A has the candidate profile B selected and it is not the case anymore that the candidate for A is bound and it is not the case anymore that a reviewer is assigned to A and it is not the case anymore that A has provisional approval and it is not the case anymore that A has a pending escalation and it is not the case anymore that A has a finalized approval and it is not the case anymore that A requires additional review

  bind a profile source B to a case instance A
    I can only bind a profile source B to a case instance A if it is the case that A is registered and B is available
    Once I bind a profile source B to a case instance A, it is the case that A has profile source B
    Once I bind a profile source B to a case instance A, it is not the case anymore that B is available

  unbind a profile source B from a case instance A
    I can only unbind a profile source B from a case instance A if it is the case that A has profile source B
    Once I unbind a profile source B from a case instance A, it is the case that B is available
    Once I unbind a profile source B from a case instance A, it is not the case anymore that A has profile source B

  assign provisional reviewer to a case instance A using a profile source B
    I can only assign provisional reviewer to a case instance A using a profile source B if it is the case that A is registered and the candidate for A is bound and A has profile source B
    I can only assign provisional reviewer to a case instance A using a profile source B if it is not the case that a reviewer is assigned to A
    Once I assign provisional reviewer to a case instance A using a profile source B, it is the case that a reviewer is assigned to A

  assign reviewer to a case instance A with an escalation ticket B
    I can only assign reviewer to a case instance A with an escalation ticket B if it is the case that A is registered and the candidate for A is bound and B is available
    I can only assign reviewer to a case instance A with an escalation ticket B if it is not the case that a reviewer is assigned to A
    Once I assign reviewer to a case instance A with an escalation ticket B, it is the case that a reviewer is assigned to A and A has a pending escalation
    Once I assign reviewer to a case instance A with an escalation ticket B, it is not the case anymore that B is available

  record provisional approval for a case instance A by a senior reviewer C with a profile source B
    I can only record provisional approval for a case instance A by a senior reviewer C with a profile source B if it is the case that a reviewer is assigned to A and the candidate for A is bound and A has profile source B and C is available as a senior reviewer
    I can only record provisional approval for a case instance A by a senior reviewer C with a profile source B if it is not the case that A has provisional approval
    Once I record provisional approval for a case instance A by a senior reviewer C with a profile source B, it is the case that A has provisional approval
    Once I record provisional approval for a case instance A by a senior reviewer C with a profile source B, it is not the case anymore that A has a pending escalation

  apply ruleset for provisional approval to a case instance A using an approval policy B
    I can only apply ruleset for provisional approval to a case instance A using an approval policy B if it is the case that the candidate for A is bound and A has engaged the approval policy B
    I can only apply ruleset for provisional approval to a case instance A using an approval policy B if it is not the case that A has provisional approval
    Once I apply ruleset for provisional approval to a case instance A using an approval policy B, it is the case that A has provisional approval
    Once I apply ruleset for provisional approval to a case instance A using an approval policy B, it is not the case anymore that A has a pending escalation

  link a case instance A to a customer segment B
    I can only link a case instance A to a customer segment B if it is the case that A is registered and B is available and A is compatible with B
    Once I link a case instance A to a customer segment B, it is the case that A has customer segment B
    Once I link a case instance A to a customer segment B, it is not the case anymore that B is available

  release a customer segment B from a case instance A
    I can only release a customer segment B from a case instance A if it is the case that A has customer segment B
    Once I release a customer segment B from a case instance A, it is the case that B is available
    Once I release a customer segment B from a case instance A, it is not the case anymore that A has customer segment B

  link an account attribute B to a case instance A
    I can only link an account attribute B to a case instance A if it is the case that A is registered and B is available and A is compatible with B
    Once I link an account attribute B to a case instance A, it is the case that A has account attribute B
    Once I link an account attribute B to a case instance A, it is not the case anymore that B is available

  release an account attribute B from a case instance A
    I can only release an account attribute B from a case instance A if it is the case that A has account attribute B
    Once I release an account attribute B from a case instance A, it is the case that B is available
    Once I release an account attribute B from a case instance A, it is not the case anymore that A has account attribute B

  link a transaction attribute B to a case instance A
    I can only link a transaction attribute B to a case instance A if it is the case that A is registered and B is available and A is compatible with B
    Once I link a transaction attribute B to a case instance A, it is the case that A has transaction attribute B
    Once I link a transaction attribute B to a case instance A, it is not the case anymore that B is available

  release a transaction attribute B from a case instance A
    I can only release a transaction attribute B from a case instance A if it is the case that A has transaction attribute B
    Once I release a transaction attribute B from a case instance A, it is the case that B is available
    Once I release a transaction attribute B from a case instance A, it is not the case anymore that A has transaction attribute B

  link a jurisdiction code B to a case instance A
    I can only link a jurisdiction code B to a case instance A if it is the case that A is registered and B is available as a jurisdiction code and A is compatible with jurisdiction code B
    Once I link a jurisdiction code B to a case instance A, it is the case that A has jurisdiction code B
    Once I link a jurisdiction code B to a case instance A, it is not the case anymore that B is available as a jurisdiction code

  release a jurisdiction code B from a case instance A
    I can only release a jurisdiction code B from a case instance A if it is the case that A has jurisdiction code B
    Once I release a jurisdiction code B from a case instance A, it is the case that B is available as a jurisdiction code
    Once I release a jurisdiction code B from a case instance A, it is not the case anymore that A has jurisdiction code B

  apply a rule set B to a case instance A with a customer segment C and an account attribute D and set gate
    I can only apply a rule set B to a case instance A with a customer segment C and an account attribute D and set gate if it is the case that the candidate for A is bound and A has customer segment C and A has account attribute D and B is available and A is compatible with B
    I can only apply a rule set B to a case instance A with a customer segment C and an account attribute D and set gate if it is not the case that A has case rules applied flag
    Once I apply a rule set B to a case instance A with a customer segment C and an account attribute D and set gate, it is the case that A has rule set B applied and A has case rules applied flag
    Once I apply a rule set B to a case instance A with a customer segment C and an account attribute D and set gate, it is not the case anymore that B is available

  apply a rule set B to a case instance A with a transaction attribute C and a subject matter expert D and set gates
    I can only apply a rule set B to a case instance A with a transaction attribute C and a subject matter expert D and set gates if it is the case that the candidate for A is bound and A has transaction attribute C and D is available as a subject matter expert and B is available and A is compatible with B and A is compatible with D
    I can only apply a rule set B to a case instance A with a transaction attribute C and a subject matter expert D and set gates if it is not the case that A has case rules applied flag
    Once I apply a rule set B to a case instance A with a transaction attribute C and a subject matter expert D and set gates, it is the case that A has rule set B applied and A has case rules applied flag and A requires additional review and A has a pending escalation
    Once I apply a rule set B to a case instance A with a transaction attribute C and a subject matter expert D and set gates, it is not the case anymore that B is available and it is not the case anymore that D is available as a subject matter expert

  clear additional review marker for a case instance A with a customer segment B and an account attribute C
    I can only clear additional review marker for a case instance A with a customer segment B and an account attribute C if it is the case that A has case rules applied flag and A requires additional review and A has customer segment B and A has account attribute C
    Once I clear additional review marker for a case instance A with a customer segment B and an account attribute C, it is not the case anymore that A requires additional review and it is not the case anymore that A has a pending escalation

  escalate a case instance A with group approval for a customer segment B, an account attribute C, and an escalation group D
    I can only escalate a case instance A with group approval for a customer segment B, an account attribute C, and an escalation group D if it is the case that A has provisional approval and A has case rules applied flag and A has customer segment B and A has account attribute C and A is handled by escalation handle D
    I can only escalate a case instance A with group approval for a customer segment B, an account attribute C, and an escalation group D if it is not the case that A has a pending escalation and it is not the case that A has case approval initiated
    Once I escalate a case instance A with group approval for a customer segment B, an account attribute C, and an escalation group D, it is the case that A has case approval initiated

  escalate a case instance A with team and SME approval for a transaction attribute B, a jurisdiction code C, and an escalation team D
    I can only escalate a case instance A with team and SME approval for a transaction attribute B, a jurisdiction code C, and an escalation team D if it is the case that A has provisional approval and A has case rules applied flag and A has transaction attribute B and A has jurisdiction code C and A is handled by escalation handle D
    I can only escalate a case instance A with team and SME approval for a transaction attribute B, a jurisdiction code C, and an escalation team D if it is not the case that A has case approval initiated
    Once I escalate a case instance A with team and SME approval for a transaction attribute B, a jurisdiction code C, and an escalation team D, it is the case that A has case approval initiated and A has a pending escalation

  finalize multifactor approval for a case instance A with a customer segment B and an account attribute C
    I can only finalize multifactor approval for a case instance A with a customer segment B and an account attribute C if it is the case that A has case approval initiated and A has a pending escalation and A has customer segment B and A has account attribute C
    Once I finalize multifactor approval for a case instance A with a customer segment B and an account attribute C, it is the case that A has a finalized approval
    Once I finalize multifactor approval for a case instance A with a customer segment B and an account attribute C, it is not the case anymore that A has a pending escalation and it is not the case anymore that A has provisional approval and it is not the case anymore that A requires additional review

  reinstate provisional approval for a case instance A with a profile source B by a senior reviewer C
    I can only reinstate provisional approval for a case instance A with a profile source B by a senior reviewer C if it is the case that A has a finalized approval and A has case approval initiated and the candidate for A is bound and A has profile source B and C is available as a senior reviewer
    I can only reinstate provisional approval for a case instance A with a profile source B by a senior reviewer C if it is not the case that A has provisional approval

  mark a profile source B reviewer required for a case instance A

    I can only mark a profile source B reviewer required for a case instance A if it is not the case that A has a pending escalation and it is not the case that A has been marked as a policy candidate
    Once I mark a profile source B reviewer required for a case instance A, it is the case that A has been marked as a policy candidate

  mark reviewer requirement for a case instance A with an escalation ticket B
    I can only mark reviewer requirement for a case instance A with an escalation ticket B if it is the case that A has case approval initiated and A has provisional approval and B is available

    Once I mark reviewer requirement for a case instance A with an escalation ticket B, it is not the case anymore that B is available

  assign an approval policy B to a case instance A

    Once I assign an approval policy B to a case instance A, it is the case that A is assigned to approval policy B
    Once I assign an approval policy B to a case instance A, it is not the case anymore that B is unassigned

  engage an approval policy C for a case category b A with a case category a case category a B
    I can only engage an approval policy C for a case category b A with a case category a case category a B if it is the case that A is registered and A is in category b and A is linked to the policy C and B is assigned to approval policy C
    I can only engage an approval policy C for a case category b A with a case category a case category a B if it is not the case that A has engaged the approval policy C
    Once I engage an approval policy C for a case category b A with a case category a case category a B, it is the case that A has engaged the approval policy C

  record senior review for a case instance A using a profile source B by a senior reviewer C
    I can only record senior review for a case instance A using a profile source B by a senior reviewer C if it is the case that A is registered and A is in category b and A has provisional approval and A has been marked as a policy candidate and A has profile source B and C is available as a senior reviewer
    I can only record senior review for a case instance A using a profile source B by a senior reviewer C if it is not the case that the senior review for A has been recorded
    Once I record senior review for a case instance A using a profile source B by a senior reviewer C, it is the case that the senior review for A has been recorded

  commit profile switch for a case instance A in category a

    I can only commit profile switch for a case instance A in category a if it is not the case that A has committed a profile switch
    Once I commit profile switch for a case instance A in category a, it is the case that A has committed a profile switch

  commit profile switch for a case instance A using an approval policy B in category b
    I can only commit profile switch for a case instance A using an approval policy B in category b if it is the case that A is in category b and A is registered and the candidate for A is bound and A has case rules applied flag and A has case approval initiated and A has been marked as a policy candidate and A has provisional approval and A has engaged the approval policy B
    I can only commit profile switch for a case instance A using an approval policy B in category b if it is not the case that A has committed a profile switch
    Once I commit profile switch for a case instance A using an approval policy B in category b, it is the case that A has committed a profile switch

  commit profile switch for a case instance A with recorded senior review in category b
    I can only commit profile switch for a case instance A with recorded senior review in category b if it is the case that A is in category b and A is registered and the candidate for A is bound and A has case rules applied flag and A has case approval initiated and A has been marked as a policy candidate and A has provisional approval and the senior review for A has been recorded
    I can only commit profile switch for a case instance A with recorded senior review in category b if it is not the case that A has committed a profile switch
    Once I commit profile switch for a case instance A with recorded senior review in category b, it is the case that A has committed a profile switch
