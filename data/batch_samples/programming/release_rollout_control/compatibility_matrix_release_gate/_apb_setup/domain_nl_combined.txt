--- TYPE HIERARCHY ---

  Everything that is a release or a deployment wave or a target environment or a dependent service or a service component or a feature toggle or a deployment group or a validation artifact or a approval policy or a validation suite or a migration job or a integration bundle or a infrastructure profile is also a object
  Everything that is a operator role or a stakeholder role is also a infrastructure profile
  Everything that is a release variant or a release anchor is also a release

--- ACTIONS ---

  register a release A as a registered release
    I can only register a release A as a registered release if it is not the case that A is a registered release and it is not the case that A has been finalized for release
    Once I register a release A as a registered release, it is the case that A is a registered release

  assign a release A to a deployment wave B
    I can only assign a release A to a deployment wave B if it is the case that A is a registered release and B is available for deployment and A is compatible with the deployment wave B
    I can only assign a release A to a deployment wave B if it is not the case that A is enrolled in the release wave
    Once I assign a release A to a deployment wave B, it is the case that A is assigned to deployment wave B and A is enrolled in the release wave
    Once I assign a release A to a deployment wave B, it is not the case anymore that B is available for deployment

  revoke a release A from a deployment wave B
    I can only revoke a release A from a deployment wave B if it is the case that A is assigned to deployment wave B
    I can only revoke a release A from a deployment wave B if it is not the case that A is promotable and it is not the case that A is approved
    Once I revoke a release A from a deployment wave B, it is the case that B is available for deployment
    Once I revoke a release A from a deployment wave B, it is not the case anymore that A is assigned to deployment wave B and it is not the case anymore that A is enrolled in the release wave and it is not the case anymore that A is prepared for the release stage and it is not the case anymore that A is validated for release and it is not the case anymore that A requires migration before release and it is not the case anymore that A has been promoted and it is not the case anymore that A has pending checks

  bind a release A to a deployment group B
    I can only bind a release A to a deployment group B if it is the case that A is a registered release and B is available
    Once I bind a release A to a deployment group B, it is the case that A is bound to the deployment group B
    Once I bind a release A to a deployment group B, it is not the case anymore that B is available

  unbind a release A from a deployment group B
    I can only unbind a release A from a deployment group B if it is the case that A is bound to the deployment group B
    Once I unbind a release A from a deployment group B, it is the case that B is available
    Once I unbind a release A from a deployment group B, it is not the case anymore that A is bound to the deployment group B

  prepare stage for a release A in a deployment group B
    I can only prepare stage for a release A in a deployment group B if it is the case that A is a registered release and A is enrolled in the release wave and A is bound to the deployment group B
    I can only prepare stage for a release A in a deployment group B if it is not the case that A is prepared for the release stage
    Once I prepare stage for a release A in a deployment group B, it is the case that A is prepared for the release stage

  prepare stage for a release A using a validation artifact B
    I can only prepare stage for a release A using a validation artifact B if it is the case that A is a registered release and A is enrolled in the release wave and B is available
    I can only prepare stage for a release A using a validation artifact B if it is not the case that A is prepared for the release stage
    Once I prepare stage for a release A using a validation artifact B, it is the case that A is prepared for the release stage and A requires migration before release
    Once I prepare stage for a release A using a validation artifact B, it is not the case anymore that B is available

  run a validation suite C for a deployment group B on a release A
    I can only run a validation suite C for a deployment group B on a release A if it is the case that A is prepared for the release stage and A is enrolled in the release wave and A is bound to the deployment group B and C is available
    I can only run a validation suite C for a deployment group B on a release A if it is not the case that A is validated for release
    Once I run a validation suite C for a deployment group B on a release A, it is the case that A is validated for release
    Once I run a validation suite C for a deployment group B on a release A, it is not the case anymore that A requires migration before release

  run validation for a release A using an approval policy B
    I can only run validation for a release A using an approval policy B if it is the case that A is enrolled in the release wave and A has the release policy applied from B
    I can only run validation for a release A using an approval policy B if it is not the case that A is validated for release
    Once I run validation for a release A using an approval policy B, it is the case that A is validated for release
    Once I run validation for a release A using an approval policy B, it is not the case anymore that A requires migration before release

  enroll a release A with component dependency a service component B
    I can only enroll a release A with component dependency a service component B if it is the case that A is a registered release and B is available and A is compatible with the service component B
    Once I enroll a release A with component dependency a service component B, it is the case that A is bound to the component B
    Once I enroll a release A with component dependency a service component B, it is not the case anymore that B is available

  unenroll a release A from component dependency a service component B

    Once I unenroll a release A from component dependency a service component B, it is the case that B is available

  enroll a release A with toggle dependency a feature toggle B

    Once I enroll a release A with toggle dependency a feature toggle B, it is not the case anymore that B is available

    Once I unenroll a release A from toggle dependency a feature toggle B, it is the case that B is available

    I can only enroll a release A in integration dependency an integration bundle B if it is the case that A is a registered release and B is available and A is compatible with the integration bundle B

    Once I unenroll a release A from integration dependency an integration bundle B, it is the case that B is available
    Once I unenroll a release A from integration dependency an integration bundle B, it is not the case anymore that A is bound to the integration bundle B

  execute integration validation for a release A in a target environment B with a service component C and a dependent service D
    I can only execute integration validation for a release A in a target environment B with a service component C and a dependent service D if it is the case that A is enrolled in the release wave and A is bound to the component C and A is bound to the service D and B is available and A is compatible with the target environment B

    Once I execute integration validation for a release A in a target environment B with a service component C and a dependent service D, it is the case that A is targeted for the environment B and A is promotable
    Once I execute integration validation for a release A in a target environment B with a service component C and a dependent service D, it is not the case anymore that B is available

    I can only execute integration with migration for a release A in a target environment B using a feature toggle C and a migration job D if it is not the case that A is promotable

    Once I clear pending checks for a release A with a service component B and a dependent service C, it is not the case anymore that A has pending checks and it is not the case anymore that A requires migration before release

    I can only mark a release A as approved by an operator role D for a service component B and a dependent service C if it is the case that A is validated for release and A is promotable and A is bound to the component B and A is bound to the service C and A is applicable to the infrastructure profile D
    I can only mark a release A as approved by an operator role D for a service component B and a dependent service C if it is not the case that A requires migration before release and it is not the case that A is approved

    I can only promote a release A after approval for a service component B and a dependent service C if it is the case that A is approved and A requires migration before release and A is bound to the component B and A is bound to the service C
    Once I promote a release A after approval for a service component B and a dependent service C, it is the case that A has been promoted
    Once I promote a release A after approval for a service component B and a dependent service C, it is not the case anymore that A requires migration before release and it is not the case anymore that A is validated for release and it is not the case anymore that A has pending checks

  mark a release A as ready for rollout in a deployment group B using a validation suite C

  finalize a release A without policy

    I can only finalize a release A with an approval policy B if it is not the case that A has been finalized for release

    I can only finalize a release A with variant ready if it is not the case that A has been finalized for release
    Once I finalize a release A with variant ready, it is the case that A has been finalized for release
