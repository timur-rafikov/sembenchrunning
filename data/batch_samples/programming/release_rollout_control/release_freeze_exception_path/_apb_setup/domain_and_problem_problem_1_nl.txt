--- ACTIONS ---

  register a release A as active
    I can only register a release A as active if it is not the case that A is active and it is not the case that A has cleared the freeze exception
    Once I register a release A as active, it is the case that A is active

  reserve a release A for a promotion wave B
    I can only reserve a release A for a promotion wave B if it is the case that A is active and the promotion wave B is available and A is eligible for release in B
    I can only reserve a release A for a promotion wave B if it is not the case that A has a reservation
    Once I reserve a release A for a promotion wave B, it is the case that A is reserved for the promotion wave B and A has a reservation
    Once I reserve a release A for a promotion wave B, it is not the case anymore that the promotion wave B is available

  unreserve a release A from a promotion wave B
    I can only unreserve a release A from a promotion wave B if it is the case that A is reserved for the promotion wave B
    I can only unreserve a release A from a promotion wave B if it is not the case that A deployment has been executed and it is not the case that final checks for A are ready
    Once I unreserve a release A from a promotion wave B, it is the case that the promotion wave B is available
    Once I unreserve a release A from a promotion wave B, it is not the case anymore that A is reserved for the promotion wave B and it is not the case anymore that A has a reservation and it is not the case anymore that A is acknowledged and it is not the case anymore that A is prevalidated and it is not the case anymore that A has been consumed by the release approver and it is not the case anymore that final approval has been granted for A and it is not the case anymore that monitoring is attached to A

  register a release A with a notification channel B
    I can only register a release A with a notification channel B if it is the case that A is active and the notification channel B is available
    Once I register a release A with a notification channel B, it is the case that A is registered with the notification channel B
    Once I register a release A with a notification channel B, it is not the case anymore that the notification channel B is available

  unregister a release A from a notification channel B
    I can only unregister a release A from a notification channel B if it is the case that A is registered with the notification channel B
    Once I unregister a release A from a notification channel B, it is the case that the notification channel B is available
    Once I unregister a release A from a notification channel B, it is not the case anymore that A is registered with the notification channel B

  acknowledge a release A on a notification channel B
    I can only acknowledge a release A on a notification channel B if it is the case that A is active and A has a reservation and A is registered with the notification channel B
    I can only acknowledge a release A on a notification channel B if it is not the case that A is acknowledged
    Once I acknowledge a release A on a notification channel B, it is the case that A is acknowledged

  record acknowledgement of a release A by an approver B
    I can only record acknowledgement of a release A by an approver B if it is the case that A is active and A has a reservation and the approver B is available
    I can only record acknowledgement of a release A by an approver B if it is not the case that A is acknowledged
    Once I record acknowledgement of a release A by an approver B, it is the case that A is acknowledged and A has been consumed by the release approver
    Once I record acknowledgement of a release A by an approver B, it is not the case anymore that the approver B is available

  apply a qa check C to a release A using a notification channel B
    I can only apply a qa check C to a release A using a notification channel B if it is the case that A is acknowledged and A has a reservation and A is registered with the notification channel B and the qa check C is available
    I can only apply a qa check C to a release A using a notification channel B if it is not the case that A is prevalidated
    Once I apply a qa check C to a release A using a notification channel B, it is the case that A is prevalidated
    Once I apply a qa check C to a release A using a notification channel B, it is not the case anymore that A has been consumed by the release approver

  bind an artifact build B to a release A
    I can only bind an artifact build B to a release A if it is the case that A has a reservation and A has confirmed artifact pinning with B
    I can only bind an artifact build B to a release A if it is not the case that A is prevalidated
    Once I bind an artifact build B to a release A, it is the case that A is prevalidated
    Once I bind an artifact build B to a release A, it is not the case anymore that A has been consumed by the release approver

  bind a service component B to a release A
    I can only bind a service component B to a release A if it is the case that A is active and the component B is available and A requires the release component B
    Once I bind a service component B to a release A, it is the case that A is bound to the service component B
    Once I bind a service component B to a release A, it is not the case anymore that the component B is available

  unbind a service component B from a release A
    I can only unbind a service component B from a release A if it is the case that A is bound to the service component B
    Once I unbind a service component B from a release A, it is the case that the component B is available
    Once I unbind a service component B from a release A, it is not the case anymore that A is bound to the service component B

  bind a configuration item B to a release A
    I can only bind a configuration item B to a release A if it is the case that A is active and the configuration item B is available and A requires the release configuration item B

    Once I unbind an integration dependency B from a release A, it is the case that the integration dependency B is available
    Once I unbind an integration dependency B from a release A, it is not the case anymore that A is bound to the integration dependency B

  schedule deployment of a release A to an environment B with a service component C and a configuration item D
    I can only schedule deployment of a release A to an environment B with a service component C and a configuration item D if it is the case that A has a reservation and A is bound to the service component C and A is bound to the configuration item D and the environment B is available and A targets the release environment B
    I can only schedule deployment of a release A to an environment B with a service component C and a configuration item D if it is not the case that A deployment has been executed
    Once I schedule deployment of a release A to an environment B with a service component C and a configuration item D, it is the case that A deployment is scheduled for B and A deployment has been executed
    Once I schedule deployment of a release A to an environment B with a service component C and a configuration item D, it is not the case anymore that the environment B is available

  execute deployment of a release A to an environment B with an integration dependency C and a monitoring profile D
    I can only execute deployment of a release A to an environment B with an integration dependency C and a monitoring profile D if it is the case that A has a reservation and A is bound to the integration dependency C and the monitoring profile D is available and the environment B is available and A targets the release environment B and A requires the monitoring profile D
    I can only execute deployment of a release A to an environment B with an integration dependency C and a monitoring profile D if it is not the case that A deployment has been executed
    Once I execute deployment of a release A to an environment B with an integration dependency C and a monitoring profile D, it is the case that A deployment is scheduled for B and A deployment has been executed and monitoring is attached to A and A has been consumed by the release approver
    Once I execute deployment of a release A to an environment B with an integration dependency C and a monitoring profile D, it is not the case anymore that the environment B is available and it is not the case anymore that the monitoring profile D is available

    Once I finalize deployment of a release A with a service component B and a configuration item C, it is not the case anymore that monitoring is attached to A and it is not the case anymore that A has been consumed by the release approver

  run final validation for a release A with a service component B, a configuration item C, and an oncall roster D
    I can only run final validation for a release A with a service component B, a configuration item C, and an oncall roster D if it is the case that A is prevalidated and A deployment has been executed and A is bound to the service component B and A is bound to the configuration item C and A is assigned to the team role D
    I can only run final validation for a release A with a service component B, a configuration item C, and an oncall roster D if it is not the case that A has been consumed by the release approver and it is not the case that final checks for A are ready
    Once I run final validation for a release A with a service component B, a configuration item C, and an oncall roster D, it is the case that final checks for A are ready

  run final validation for a release A with an integration dependency B, a dependency ticket C, and an incident response roster D
    I can only run final validation for a release A with an integration dependency B, a dependency ticket C, and an incident response roster D if it is the case that A is prevalidated and A deployment has been executed and A is bound to the integration dependency B and A is bound to the dependency ticket C and A is assigned to the team role D
    I can only run final validation for a release A with an integration dependency B, a dependency ticket C, and an incident response roster D if it is not the case that final checks for A are ready

    Once I apply final approval to a release A with a service component B and a configuration item C, it is the case that final approval has been granted for A
    Once I apply final approval to a release A with a service component B and a configuration item C, it is not the case anymore that A has been consumed by the release approver and it is not the case anymore that A is prevalidated and it is not the case anymore that monitoring is attached to A

    Once I record cross release constraint for a release A using a notification channel B with a qa check C, it is the case that cross release constraints have been recorded for A

    I can only mark a release A as cleared with artifact pin an artifact build B if it is not the case that A has cleared the freeze exception
