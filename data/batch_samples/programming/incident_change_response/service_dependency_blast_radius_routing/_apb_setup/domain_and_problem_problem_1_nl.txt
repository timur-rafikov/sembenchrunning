--- ACTIONS ---

  create change case for a service case A
    I can only create change case for a service case A if it is not the case that A is open and it is not the case that the case for A is closed
    Once I create change case for a service case A, it is the case that A is open

  assign an engineer B to a service case A
    I can only assign an engineer B to a service case A if it is the case that A is open and B is available and B is eligible for the case A
    I can only assign an engineer B to a service case A if it is not the case that the assignee is reserved for A
    Once I assign an engineer B to a service case A, it is the case that A is assigned to B and the assignee is reserved for A
    Once I assign an engineer B to a service case A, it is not the case anymore that B is available

  release an engineer B from a service case A
    I can only release an engineer B from a service case A if it is the case that A is assigned to B
    I can only release an engineer B from a service case A if it is not the case that a mitigation is authorized for A and it is not the case that approvals have been collected for A
    Once I release an engineer B from a service case A, it is the case that B is available
    Once I release an engineer B from a service case A, it is not the case anymore that A is assigned to B and it is not the case anymore that the assignee is reserved for A and it is not the case anymore that preapproval is obtained for A and it is not the case anymore that route prechecks have passed for A and it is not the case anymore that stakeholder acknowledgment is required for A and it is not the case anymore that execution is committed for A and it is not the case anymore that external review is required for A

  reserve a routing token B for a service case A
    I can only reserve a routing token B for a service case A if it is the case that A is open and B is available for routing
    Once I reserve a routing token B for a service case A, it is the case that B is reserved for A
    Once I reserve a routing token B for a service case A, it is not the case anymore that B is available for routing

  release a routing token B from a service case A

    Once I release a routing token B from a service case A, it is the case that B is available for routing
    Once I release a routing token B from a service case A, it is not the case anymore that B is reserved for A

  record preapproval for a service case A using a routing token B
    I can only record preapproval for a service case A using a routing token B if it is the case that A is open and the assignee is reserved for A and B is reserved for A
    I can only record preapproval for a service case A using a routing token B if it is not the case that preapproval is obtained for A
    Once I record preapproval for a service case A using a routing token B, it is the case that preapproval is obtained for A

  obtain preapproval for a service case A from an approver B

    I can only obtain preapproval for a service case A from an approver B if it is not the case that preapproval is obtained for A
    Once I obtain preapproval for a service case A from an approver B, it is the case that preapproval is obtained for A and stakeholder acknowledgment is required for A
    Once I obtain preapproval for a service case A from an approver B, it is not the case anymore that B is available

  bind a network control C with a routing token B and mark a service case A as ready
    I can only bind a network control C with a routing token B and mark a service case A as ready if it is the case that preapproval is obtained for A and the assignee is reserved for A and B is reserved for A and C is available
    I can only bind a network control C with a routing token B and mark a service case A as ready if it is not the case that route prechecks have passed for A

    Once I bind a network control C with a routing token B and mark a service case A as ready, it is not the case anymore that stakeholder acknowledgment is required for A

  use a mitigation action B to mark a service case A as ready
    I can only use a mitigation action B to mark a service case A as ready if it is the case that the assignee is reserved for A and B is bound to the A instance
    I can only use a mitigation action B to mark a service case A as ready if it is not the case that route prechecks have passed for A
    Once I use a mitigation action B to mark a service case A as ready, it is the case that route prechecks have passed for A
    Once I use a mitigation action B to mark a service case A as ready, it is not the case anymore that stakeholder acknowledgment is required for A

  reserve an upstream service B for a service case A
    I can only reserve an upstream service B for a service case A if it is the case that A is open and B is available upstream and B is a candidate for the case A
