--- TYPE HIERARCHY ---

  Everything that is a change incident case or a oncall engineer or a deployment window or a system component or a workaround mitigation or a operational action or a runbook or a approver person or a fix artifact or a executor operator or a test environment or a rollback artifact or a stakeholder group is also a object
  Everything that is a approval channel or a escalation channel is also a stakeholder group
  Everything that is a case alias primary or a case alias secondary is also a change incident case

--- ACTIONS ---

  open a change incident case A
    I can only open a change incident case A if it is not the case that the case A is active and it is not the case that A is closed
    Once I open a change incident case A, it is the case that the case A is active

  assign an oncall engineer B to a change incident case A as the responder
    I can only assign an oncall engineer B to a change incident case A as the responder if it is the case that the case A is active and B is available as an engineer and B is eligible for the case A
    I can only assign an oncall engineer B to a change incident case A as the responder if it is not the case that a responder is allocated to the case A
    Once I assign an oncall engineer B to a change incident case A as the responder, it is the case that B is the assigned responder for the case A and a responder is allocated to the case A
    Once I assign an oncall engineer B to a change incident case A as the responder, it is not the case anymore that B is available as an engineer

  unassign an oncall engineer B from a change incident case A as the responder
    I can only unassign an oncall engineer B from a change incident case A as the responder if it is the case that B is the assigned responder for the case A
    I can only unassign an oncall engineer B from a change incident case A as the responder if it is not the case that A is ready to be fixed and it is not the case that approval has been granted for A
    Once I unassign an oncall engineer B from a change incident case A as the responder, it is the case that B is available as an engineer
    Once I unassign an oncall engineer B from a change incident case A as the responder, it is not the case anymore that B is the assigned responder for the case A and it is not the case anymore that a responder is allocated to the case A and it is not the case anymore that mitigation is authorized for the case A and it is not the case anymore that remediation is applied to A and it is not the case anymore that A requires escalation and it is not the case anymore that the execution order for A is finalized and it is not the case anymore that A requires additional review

  bind a runbook B to a change incident case A
    I can only bind a runbook B to a change incident case A if it is the case that the case A is active and B is an available runbook
    Once I bind a runbook B to a change incident case A, it is the case that B is bound to the case A
    Once I bind a runbook B to a change incident case A, it is not the case anymore that B is an available runbook

  unbind a runbook B from a change incident case A
    I can only unbind a runbook B from a change incident case A if it is the case that B is bound to the case A
    Once I unbind a runbook B from a change incident case A, it is the case that B is an available runbook
    Once I unbind a runbook B from a change incident case A, it is not the case anymore that B is bound to the case A

  authorize execution of a runbook B for a change incident case A
    I can only authorize execution of a runbook B for a change incident case A if it is the case that the case A is active and a responder is allocated to the case A and B is bound to the case A
    I can only authorize execution of a runbook B for a change incident case A if it is not the case that mitigation is authorized for the case A
    Once I authorize execution of a runbook B for a change incident case A, it is the case that mitigation is authorized for the case A

  approve mitigation for a change incident case A by an approver person B
    I can only approve mitigation for a change incident case A by an approver person B if it is the case that the case A is active and a responder is allocated to the case A and B is an available approver
    I can only approve mitigation for a change incident case A by an approver person B if it is not the case that mitigation is authorized for the case A
    Once I approve mitigation for a change incident case A by an approver person B, it is the case that mitigation is authorized for the case A and A requires escalation
    Once I approve mitigation for a change incident case A by an approver person B, it is not the case anymore that B is an available approver

  execute a runbook B for a change incident case A by an executor operator C
    I can only execute a runbook B for a change incident case A by an executor operator C if it is the case that mitigation is authorized for the case A and a responder is allocated to the case A and B is bound to the case A and C is available as an executor
    I can only execute a runbook B for a change incident case A by an executor operator C if it is not the case that remediation is applied to A
    Once I execute a runbook B for a change incident case A by an executor operator C, it is the case that remediation is applied to A
    Once I execute a runbook B for a change incident case A by an executor operator C, it is not the case anymore that A requires escalation

  apply a fix artifact B to a change incident case A
    I can only apply a fix artifact B to a change incident case A if it is the case that a responder is allocated to the case A and the fix cross case A is authorized for B
    I can only apply a fix artifact B to a change incident case A if it is not the case that remediation is applied to A
    Once I apply a fix artifact B to a change incident case A, it is the case that remediation is applied to A
    Once I apply a fix artifact B to a change incident case A, it is not the case anymore that A requires escalation

  select a workaround mitigation B for a change incident case A
    I can only select a workaround mitigation B for a change incident case A if it is the case that the case A is active and B is an available mitigation and B is applicable to the case A
    Once I select a workaround mitigation B for a change incident case A, it is the case that B is selected as mitigation for A
    Once I select a workaround mitigation B for a change incident case A, it is not the case anymore that B is an available mitigation

  deselect a workaround mitigation B from a change incident case A
    I can only deselect a workaround mitigation B from a change incident case A if it is the case that B is selected as mitigation for A
    Once I deselect a workaround mitigation B from a change incident case A, it is the case that B is an available mitigation
    Once I deselect a workaround mitigation B from a change incident case A, it is not the case anymore that B is selected as mitigation for A

  reserve a system component B for a change incident case A

  propose and schedule a fix for a change incident case A using a workaround mitigation C on a system component D during a deployment window B
    I can only propose and schedule a fix for a change incident case A using a workaround mitigation C on a system component D during a deployment window B if it is the case that a responder is allocated to the case A and C is selected as mitigation for A and D is reserved for A and B is an available window and B is eligible for the case A

    Once I propose a fix with operational validation for a change incident case A using an operational action C in a test environment D during a deployment window B, it is not the case anymore that B is an available window and it is not the case anymore that D is available for testing
