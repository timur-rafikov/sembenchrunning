============================================================
DOMAIN IN NATURAL LANGUAGE
============================================================

--- TYPE HIERARCHY ---

  Everything that is a session slot or a oncall engineer or a change request or a approver or a mitigation plan or a diagnostic check or a checklist item or a external ack or a service or a runbook or a test environment or a change package or a stakeholder role is also a object
  Everything that is a primary stakeholder or a secondary stakeholder is also a stakeholder role
  Everything that is a incoming session or a outgoing session is also a session slot

  the session {?handoff_session} is initialized
  the session {?handoff_session} is assigned to the engineer {?oncall_engineer}
  the engineer for the session {?handoff_session} is confirmed
  the checklist for the session {?handoff_session} is verified
  the runbook for the session {?handoff_session} is executed
  the session {?handoff_session} has a mitigation plan {?mitigation_plan}

  the change request for session {?handoff_session} is locked

  session {?handoff_session} is finalized

--- ACTIONS ---

  initialize handoff session for a session slot A
    I can only initialize handoff session for a session slot A if it is not the case that the session A is initialized and it is not the case that session A is finalized
    Once I initialize handoff session for a session slot A, it is the case that the session A is initialized

  assign an oncall engineer B to a session slot A
    I can only assign an oncall engineer B to a session slot A if it is the case that the session A is initialized and B is an available engineer and B is eligible for the session A
    I can only assign an oncall engineer B to a session slot A if it is not the case that the engineer for the session A is confirmed
    Once I assign an oncall engineer B to a session slot A, it is the case that the session A is assigned to the engineer B and the engineer for the session A is confirmed
    Once I assign an oncall engineer B to a session slot A, it is not the case anymore that B is an available engineer

  release an oncall engineer B from a session slot A
    I can only release an oncall engineer B from a session slot A if it is the case that the session A is assigned to the engineer B
    I can only release an oncall engineer B from a session slot A if it is not the case that the change request for session A is locked and it is not the case that authorization for session A is granted
    Once I release an oncall engineer B from a session slot A, it is the case that B is an available engineer
    Once I release an oncall engineer B from a session slot A, it is not the case anymore that the session A is assigned to the engineer B and it is not the case anymore that the engineer for the session A is confirmed and it is not the case anymore that the checklist for the session A is verified and it is not the case anymore that the runbook for the session A is executed and it is not the case anymore that session preexecution flag is set for A and it is not the case anymore that session execution is enabled for A and it is not the case anymore that A requires additional validation

  bind a checklist item B to a session slot A checklist
    I can only bind a checklist item B to a session slot A checklist if it is the case that the session A is initialized and B is a checklist item available
    Once I bind a checklist item B to a session slot A checklist, it is the case that B is a bound item in the session checklist for A
    Once I bind a checklist item B to a session slot A checklist, it is not the case anymore that B is a checklist item available

  unbind a checklist item B from a session slot A checklist
    I can only unbind a checklist item B from a session slot A checklist if it is the case that B is a bound item in the session checklist for A
    Once I unbind a checklist item B from a session slot A checklist, it is the case that B is a checklist item available
    Once I unbind a checklist item B from a session slot A checklist, it is not the case anymore that B is a bound item in the session checklist for A

  verify a checklist item B for a session slot A
    I can only verify a checklist item B for a session slot A if it is the case that the session A is initialized and the engineer for the session A is confirmed and B is a bound item in the session checklist for A
    I can only verify a checklist item B for a session slot A if it is not the case that the checklist for the session A is verified
    Once I verify a checklist item B for a session slot A, it is the case that the checklist for the session A is verified

  record an external ack B and verify for a session slot A
    I can only record an external ack B and verify for a session slot A if it is the case that the session A is initialized and the engineer for the session A is confirmed and B is an external acknowledgment available
    I can only record an external ack B and verify for a session slot A if it is not the case that the checklist for the session A is verified
    Once I record an external ack B and verify for a session slot A, it is the case that the checklist for the session A is verified and session preexecution flag is set for A
    Once I record an external ack B and verify for a session slot A, it is not the case anymore that B is an external acknowledgment available

  attach a runbook C to a checklist item B and mark for a session slot A
    I can only attach a runbook C to a checklist item B and mark for a session slot A if it is the case that the checklist for the session A is verified and the engineer for the session A is confirmed and B is a bound item in the session checklist for A and C is available
    I can only attach a runbook C to a checklist item B and mark for a session slot A if it is not the case that the runbook for the session A is executed
    Once I attach a runbook C to a checklist item B and mark for a session slot A, it is the case that the runbook for the session A is executed
    Once I attach a runbook C to a checklist item B and mark for a session slot A, it is not the case anymore that session preexecution flag is set for A

  mark runbook for a service B in a session slot A
    I can only mark runbook for a service B in a session slot A if it is the case that the engineer for the session A is confirmed and A has a session service verified with B
    I can only mark runbook for a service B in a session slot A if it is not the case that the runbook for the session A is executed
    Once I mark runbook for a service B in a session slot A, it is the case that the runbook for the session A is executed
    Once I mark runbook for a service B in a session slot A, it is not the case anymore that session preexecution flag is set for A

  reserve a mitigation plan B for a session slot A
    I can only reserve a mitigation plan B for a session slot A if it is the case that the session A is initialized and B is an available mitigation plan and B is assigned to the session A
    Once I reserve a mitigation plan B for a session slot A, it is the case that the session A has a mitigation plan B
    Once I reserve a mitigation plan B for a session slot A, it is not the case anymore that B is an available mitigation plan

  release a mitigation plan B from a session slot A
    I can only release a mitigation plan B from a session slot A if it is the case that the session A has a mitigation plan B
    Once I release a mitigation plan B from a session slot A, it is the case that B is an available mitigation plan
    Once I release a mitigation plan B from a session slot A, it is not the case anymore that the session A has a mitigation plan B

  reserve an approver B for a session slot A
    I can only reserve an approver B for a session slot A if it is the case that the session A is initialized and B is an available approver and A has an approver binding with B
    Once I reserve an approver B for a session slot A, it is the case that the session A has an approver B
    Once I reserve an approver B for a session slot A, it is not the case anymore that B is an available approver

  release an approver B from a session slot A
    I can only release an approver B from a session slot A if it is the case that the session A has an approver B
    Once I release an approver B from a session slot A, it is the case that B is an available approver
    Once I release an approver B from a session slot A, it is not the case anymore that the session A has an approver B

  reserve a diagnostic check B for a session slot A
    I can only reserve a diagnostic check B for a session slot A if it is the case that the session A is initialized and B is a diagnostic available and A has a diagnostic binding with B
    Once I reserve a diagnostic check B for a session slot A, it is the case that the session A has a diagnostic check B
    Once I reserve a diagnostic check B for a session slot A, it is not the case anymore that B is a diagnostic available

  release a diagnostic check B from a session slot A
    I can only release a diagnostic check B from a session slot A if it is the case that the session A has a diagnostic check B
    Once I release a diagnostic check B from a session slot A, it is the case that B is a diagnostic available
    Once I release a diagnostic check B from a session slot A, it is not the case anymore that the session A has a diagnostic check B

  reserve a change package B for a session slot A
    I can only reserve a change package B for a session slot A if it is the case that the session A is initialized and B is available and A has a change package binding with B
    Once I reserve a change package B for a session slot A, it is the case that the session A has a change package B
    Once I reserve a change package B for a session slot A, it is not the case anymore that B is available

  release a change package B from a session slot A
    I can only release a change package B from a session slot A if it is the case that the session A has a change package B
    Once I release a change package B from a session slot A, it is the case that B is available
    Once I release a change package B from a session slot A, it is not the case anymore that the session A has a change package B

  bind a change request B with required artifacts in a session slot A using a mitigation plan C and an approver D
    I can only bind a change request B with required artifacts in a session slot A using a mitigation plan C and an approver D if it is the case that the engineer for the session A is confirmed and the session A has a mitigation plan C and the session A has an approver D and B is a valid change request and B is linked to the session A
    I can only bind a change request B with required artifacts in a session slot A using a mitigation plan C and an approver D if it is not the case that the change request for session A is locked
    Once I bind a change request B with required artifacts in a session slot A using a mitigation plan C and an approver D, it is the case that the session A has a change request B ready and the change request for session A is locked
    Once I bind a change request B with required artifacts in a session slot A using a mitigation plan C and an approver D, it is not the case anymore that B is a valid change request

  bind a change request B with diagnostic and test environment in a session slot A using a diagnostic check C and a test environment D
    I can only bind a change request B with diagnostic and test environment in a session slot A using a diagnostic check C and a test environment D if it is the case that the engineer for the session A is confirmed and the session A has a diagnostic check C and D is available and B is a valid change request and B is linked to the session A and A has a test environment binding with D
    I can only bind a change request B with diagnostic and test environment in a session slot A using a diagnostic check C and a test environment D if it is not the case that the change request for session A is locked
    Once I bind a change request B with diagnostic and test environment in a session slot A using a diagnostic check C and a test environment D, it is the case that the session A has a change request B ready and the change request for session A is locked and A requires additional validation and session preexecution flag is set for A
    Once I bind a change request B with diagnostic and test environment in a session slot A using a diagnostic check C and a test environment D, it is not the case anymore that B is a valid change request and it is not the case anymore that D is available

  consume approvals for a session slot A using a mitigation plan B and an approver C before execution
    I can only consume approvals for a session slot A using a mitigation plan B and an approver C before execution if it is the case that the change request for session A is locked and A requires additional validation and the session A has a mitigation plan B and the session A has an approver C
    Once I consume approvals for a session slot A using a mitigation plan B and an approver C before execution, it is not the case anymore that A requires additional validation and it is not the case anymore that session preexecution flag is set for A

  record multi party authorization for a session slot A with a mitigation plan B, an approver C, and a primary stakeholder D
    I can only record multi party authorization for a session slot A with a mitigation plan B, an approver C, and a primary stakeholder D if it is the case that the runbook for the session A is executed and the change request for session A is locked and the session A has a mitigation plan B and the session A has an approver C and A requires a stakeholder with the role D
    I can only record multi party authorization for a session slot A with a mitigation plan B, an approver C, and a primary stakeholder D if it is not the case that session preexecution flag is set for A and it is not the case that authorization for session A is granted
    Once I record multi party authorization for a session slot A with a mitigation plan B, an approver C, and a primary stakeholder D, it is the case that authorization for session A is granted

  record multi party authorization with secondary for a session slot A including a diagnostic check B, a change package C, and a secondary stakeholder D
    I can only record multi party authorization with secondary for a session slot A including a diagnostic check B, a change package C, and a secondary stakeholder D if it is the case that the runbook for the session A is executed and the change request for session A is locked and the session A has a diagnostic check B and the session A has a change package C and A requires a stakeholder with the role D
    I can only record multi party authorization with secondary for a session slot A including a diagnostic check B, a change package C, and a secondary stakeholder D if it is not the case that authorization for session A is granted
    Once I record multi party authorization with secondary for a session slot A including a diagnostic check B, a change package C, and a secondary stakeholder D, it is the case that authorization for session A is granted and session preexecution flag is set for A

  finalize authorization and prepare execution for a session slot A with a mitigation plan B and an approver C
    I can only finalize authorization and prepare execution for a session slot A with a mitigation plan B and an approver C if it is the case that authorization for session A is granted and session preexecution flag is set for A and the session A has a mitigation plan B and the session A has an approver C

    Once I finalize authorization and prepare execution for a session slot A with a mitigation plan B and an approver C, it is not the case anymore that session preexecution flag is set for A and it is not the case anymore that the runbook for the session A is executed and it is not the case anymore that A requires additional validation

  perform checklist step with a runbook C for a session slot A and a checklist item B

    I can only perform checklist step with a runbook C for a session slot A and a checklist item B if it is not the case that the runbook for the session A is executed
    Once I perform checklist step with a runbook C for a session slot A and a checklist item B, it is the case that the runbook for the session A is executed

  finalize a checklist item B for a session slot A

    I can only finalize a checklist item B for a session slot A if it is not the case that session preexecution flag is set for A and it is not the case that the checklist for session A is finalized
    Once I finalize a checklist item B for a session slot A, it is the case that the checklist for session A is finalized

  finalize a session slot A with an external ack B
    I can only finalize a session slot A with an external ack B if it is the case that authorization for session A is granted and the runbook for the session A is executed and B is an external acknowledgment available
    I can only finalize a session slot A with an external ack B if it is not the case that session preexecution flag is set for A and it is not the case that the checklist for session A is finalized
    Once I finalize a session slot A with an external ack B, it is the case that the checklist for session A is finalized
    Once I finalize a session slot A with an external ack B, it is not the case anymore that B is an external acknowledgment available

  bind service execution token for a session slot A to a service B
    I can only bind service execution token for a session slot A to a service B if it is the case that the checklist for session A is finalized and B has an execution token available and A binds to the participant service B
    Once I bind service execution token for a session slot A to a service B, it is the case that A has a service execution token for B
    Once I bind service execution token for a session slot A to a service B, it is not the case anymore that B has an execution token available

  record participant transfer confirmation from an outgoing session A to an incoming session B for a service C
    I can only record participant transfer confirmation from an outgoing session A to an incoming session B for a service C if it is the case that the session A is initialized and there is an outgoing slot present in A and A has a participant outgoing service binding with C and B has a service execution token for C
    I can only record participant transfer confirmation from an outgoing session A to an incoming session B for a service C if it is not the case that A has a session service verified with C
    Once I record participant transfer confirmation from an outgoing session A to an incoming session B for a service C, it is the case that A has a session service verified with C

  record post transfer check completion for a session slot A with a checklist item B using a runbook C
    I can only record post transfer check completion for a session slot A with a checklist item B using a runbook C if it is the case that the session A is initialized and there is an outgoing slot present in A and the runbook for the session A is executed and the checklist for session A is finalized and B is a bound item in the session checklist for A and C is available
    I can only record post transfer check completion for a session slot A with a checklist item B using a runbook C if it is not the case that a session participant has been transferred in A
    Once I record post transfer check completion for a session slot A with a checklist item B using a runbook C, it is the case that a session participant has been transferred in A

  set finalization flag for a session slot A

    I can only set finalization flag for a session slot A if it is not the case that session A is finalized
    Once I set finalization flag for a session slot A, it is the case that session A is finalized

  set finalization flag for a session slot A with service verification of a service B

    I can only set finalization flag for a session slot A with service verification of a service B if it is not the case that session A is finalized
