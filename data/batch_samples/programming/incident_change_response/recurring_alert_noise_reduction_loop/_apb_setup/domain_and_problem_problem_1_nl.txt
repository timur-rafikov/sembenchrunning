  Everything that is a incident case or a responder or a mitigation runbook or a configuration change or a alert rule or a suppression rule or a alert signature or a approver or a change ticket or a automation playbook or a rollback plan or a tag or a execution context is also a object
  Everything that is a human team or a automation identity is also a execution context
  Everything that is a incident variant or a incident origin is also a incident case

--- ACTIONS ---

  open an incident case A
    I can only open an incident case A if it is not the case that A is an open incident and it is not the case that the incident A is resolved
    Once I open an incident case A, it is the case that A is an open incident

  assign a responder B to an incident case A
    I can only assign a responder B to an incident case A if it is the case that A is an open incident and B is available and B is eligible for A
    I can only assign a responder B to an incident case A if it is not the case that A has an owner
    Once I assign a responder B to an incident case A, it is the case that B is assigned to A and A has an owner
    Once I assign a responder B to an incident case A, it is not the case anymore that B is available

  unassign a responder B from an incident case A
    I can only unassign a responder B from an incident case A if it is the case that B is assigned to A
    I can only unassign a responder B from an incident case A if it is not the case that a runbook is scheduled for A and it is not the case that a change is prepared for A
    Once I unassign a responder B from an incident case A, it is the case that B is available
    Once I unassign a responder B from an incident case A, it is not the case anymore that B is assigned to A and it is not the case anymore that A has an owner and it is not the case anymore that the signature of A is escalated and it is not the case anymore that A is ready for remediation and it is not the case anymore that A is awaiting approval and it is not the case anymore that A is in implementation in progress and it is not the case anymore that a change is requested for A

  bind an alert signature B to an incident case A
    I can only bind an alert signature B to an incident case A if it is the case that A is an open incident and the alert signature B is available
    Once I bind an alert signature B to an incident case A, it is the case that the signature B is bound to the incident A
    Once I bind an alert signature B to an incident case A, it is not the case anymore that the alert signature B is available

  unbind an alert signature B from an incident case A
    I can only unbind an alert signature B from an incident case A if it is the case that the signature B is bound to the incident A
    Once I unbind an alert signature B from an incident case A, it is the case that the alert signature B is available
    Once I unbind an alert signature B from an incident case A, it is not the case anymore that the signature B is bound to the incident A

  escalate an alert signature B for an incident case A
    I can only escalate an alert signature B for an incident case A if it is the case that A is an open incident and A has an owner and the signature B is bound to the incident A
    I can only escalate an alert signature B for an incident case A if it is not the case that the signature of A is escalated
    Once I escalate an alert signature B for an incident case A, it is the case that the signature of A is escalated

  escalate an incident case A to an approver B
    I can only escalate an incident case A to an approver B if it is the case that A is an open incident and A has an owner and the approver B is available
    I can only escalate an incident case A to an approver B if it is not the case that the signature of A is escalated
    Once I escalate an incident case A to an approver B, it is the case that the signature of A is escalated and A is awaiting approval
    Once I escalate an incident case A to an approver B, it is not the case anymore that the approver B is available

  prepare an automation playbook C for an incident case A with an alert signature B
    I can only prepare an automation playbook C for an incident case A with an alert signature B if it is the case that the signature of A is escalated and A has an owner and the signature B is bound to the incident A and the playbook C is available
    I can only prepare an automation playbook C for an incident case A with an alert signature B if it is not the case that A is ready for remediation
    Once I prepare an automation playbook C for an incident case A with an alert signature B, it is the case that A is ready for remediation
    Once I prepare an automation playbook C for an incident case A with an alert signature B, it is not the case anymore that A is awaiting approval

  prepare automation for an incident case A using a change ticket B
    I can only prepare automation for an incident case A using a change ticket B if it is the case that A has an owner and A has a ticket scheduled for B
    I can only prepare automation for an incident case A using a change ticket B if it is not the case that A is ready for remediation
    Once I prepare automation for an incident case A using a change ticket B, it is the case that A is ready for remediation
    Once I prepare automation for an incident case A using a change ticket B, it is not the case anymore that A is awaiting approval

  link an alert rule B to an incident case A
    I can only link an alert rule B to an incident case A if it is the case that A is an open incident and the alert rule B is available and B is applicable for A
    Once I link an alert rule B to an incident case A, it is the case that A is linked to the alert rule B
    Once I link an alert rule B to an incident case A, it is not the case anymore that the alert rule B is available

  unlink an alert rule B from an incident case A
    I can only unlink an alert rule B from an incident case A if it is the case that A is linked to the alert rule B
    Once I unlink an alert rule B from an incident case A, it is the case that the alert rule B is available
    Once I unlink an alert rule B from an incident case A, it is not the case anymore that A is linked to the alert rule B

  link a configuration change B to an incident case A
    I can only link a configuration change B to an incident case A if it is the case that A is an open incident and the configuration change B is available and B is applicable for A
    Once I link a configuration change B to an incident case A, it is the case that A is linked to the configuration change B
    Once I link a configuration change B to an incident case A, it is not the case anymore that the configuration change B is available

  unlink a configuration change B from an incident case A
    I can only unlink a configuration change B from an incident case A if it is the case that A is linked to the configuration change B
    Once I unlink a configuration change B from an incident case A, it is the case that the configuration change B is available
    Once I unlink a configuration change B from an incident case A, it is not the case anymore that A is linked to the configuration change B

  link a suppression rule B to an incident case A
    I can only link a suppression rule B to an incident case A if it is the case that A is an open incident and the suppression rule B is available and B is applicable for A
    Once I link a suppression rule B to an incident case A, it is the case that A is linked to the suppression rule B
    Once I link a suppression rule B to an incident case A, it is not the case anymore that the suppression rule B is available

  unlink a suppression rule B from an incident case A
    I can only unlink a suppression rule B from an incident case A if it is the case that A is linked to the suppression rule B
    Once I unlink a suppression rule B from an incident case A, it is the case that the suppression rule B is available
    Once I unlink a suppression rule B from an incident case A, it is not the case anymore that A is linked to the suppression rule B

  link a tag B to an incident case A
    I can only link a tag B to an incident case A if it is the case that A is an open incident and the tag B is available and B is applicable for A
    Once I link a tag B to an incident case A, it is the case that A is linked to the tag B
    Once I link a tag B to an incident case A, it is not the case anymore that the tag B is available

  unlink a tag B from an incident case A
    I can only unlink a tag B from an incident case A if it is the case that A is linked to the tag B
    Once I unlink a tag B from an incident case A, it is the case that the tag B is available
    Once I unlink a tag B from an incident case A, it is not the case anymore that A is linked to the tag B

  generate a mitigation plan for an incident case A using a mitigation runbook B with an alert rule C and a configuration change D
    I can only generate a mitigation plan for an incident case A using a mitigation runbook B with an alert rule C and a configuration change D if it is the case that A has an owner and A is linked to the alert rule C and A is linked to the configuration change D and the runbook B is available and B is applicable for A
    I can only generate a mitigation plan for an incident case A using a mitigation runbook B with an alert rule C and a configuration change D if it is not the case that a runbook is scheduled for A
    Once I generate a mitigation plan for an incident case A using a mitigation runbook B with an alert rule C and a configuration change D, it is the case that A is linked to the mitigation runbook B and a runbook is scheduled for A
    Once I generate a mitigation plan for an incident case A using a mitigation runbook B with an alert rule C and a configuration change D, it is not the case anymore that the runbook B is available

  generate a mitigation plan with rollback for an incident case A using a mitigation runbook B, a suppression rule C, and a rollback plan D
    I can only generate a mitigation plan with rollback for an incident case A using a mitigation runbook B, a suppression rule C, and a rollback plan D if it is the case that A has an owner and A is linked to the suppression rule C and the rollback plan D is available and the runbook B is available and B is applicable for A and D is applicable for A
    I can only generate a mitigation plan with rollback for an incident case A using a mitigation runbook B, a suppression rule C, and a rollback plan D if it is not the case that a runbook is scheduled for A
    Once I generate a mitigation plan with rollback for an incident case A using a mitigation runbook B, a suppression rule C, and a rollback plan D, it is the case that A is linked to the mitigation runbook B and a runbook is scheduled for A and a change is requested for A and A is awaiting approval
    Once I generate a mitigation plan with rollback for an incident case A using a mitigation runbook B, a suppression rule C, and a rollback plan D, it is not the case anymore that the runbook B is available and it is not the case anymore that the rollback plan D is available

  validate change prerequisites for an incident case A using an alert rule B and a configuration change C
    I can only validate change prerequisites for an incident case A using an alert rule B and a configuration change C if it is the case that a runbook is scheduled for A and a change is requested for A and A is linked to the alert rule B and A is linked to the configuration change C
    Once I validate change prerequisites for an incident case A using an alert rule B and a configuration change C, it is not the case anymore that a change is requested for A and it is not the case anymore that A is awaiting approval

  record change approval for an incident case A using an alert rule B, a configuration change C, and a human team D
    I can only record change approval for an incident case A using an alert rule B, a configuration change C, and a human team D if it is the case that A is ready for remediation and a runbook is scheduled for A and A is linked to the alert rule B and A is linked to the configuration change C and A is bound to D
    I can only record change approval for an incident case A using an alert rule B, a configuration change C, and a human team D if it is not the case that A is awaiting approval and it is not the case that a change is prepared for A
    Once I record change approval for an incident case A using an alert rule B, a configuration change C, and a human team D, it is the case that a change is prepared for A

  record change approval for an incident case A using a suppression rule B, a tag C, and an automation identity D
    I can only record change approval for an incident case A using a suppression rule B, a tag C, and an automation identity D if it is the case that A is ready for remediation and a runbook is scheduled for A and A is linked to the suppression rule B and A is linked to the tag C and A is bound to D
    I can only record change approval for an incident case A using a suppression rule B, a tag C, and an automation identity D if it is not the case that a change is prepared for A
    Once I record change approval for an incident case A using a suppression rule B, a tag C, and an automation identity D, it is the case that a change is prepared for A and A is awaiting approval

  start implementation for an incident case A using an alert rule B and a configuration change C
    I can only start implementation for an incident case A using an alert rule B and a configuration change C if it is the case that a change is prepared for A and A is awaiting approval and A is linked to the alert rule B and A is linked to the configuration change C

    Once I mark the ticket for an incident case A as pending with an approver B, it is not the case anymore that the approver B is available

  bind a change ticket B to an incident case A

    Once I bind a change ticket B to an incident case A, it is the case that A is bound to B
    Once I bind a change ticket B to an incident case A, it is not the case anymore that the change ticket B is available

  schedule a change ticket C for an incident origin A with an incident variant B

    I can only schedule a change ticket C for an incident origin A with an incident variant B if it is not the case that A has a ticket scheduled for C
    Once I schedule a change ticket C for an incident origin A with an incident variant B, it is the case that A has a ticket scheduled for C

  record execution observation for an incident case A using an alert signature B and an automation playbook C
    I can only record execution observation for an incident case A using an alert signature B and an automation playbook C if it is the case that A is an open incident and the origin is verified for A and A is ready for remediation and the incident ticket for A is pending and the signature B is bound to the incident A and the playbook C is available
    I can only record execution observation for an incident case A using an alert signature B and an automation playbook C if it is not the case that A has an implementation observed
    Once I record execution observation for an incident case A using an alert signature B and an automation playbook C, it is the case that A has an implementation observed

  close incident an incident case A using standard procedure

    I can only close incident an incident case A using standard procedure if it is not the case that the incident A is resolved
    Once I close incident an incident case A using standard procedure, it is the case that the incident A is resolved

  close incident an incident case A with a change ticket B
